# Self-updater

In-app self-update. Checks GitHub Releases for the latest `build-N` tag,
compares with `longVersionCode`, downloads the APK, installs via
PackageInstaller. Triggered manually via the `Check for update` button.

## How it works (current version)

1. **Check** — HTTP HEAD-like request to `releases/latest`, follow the redirect
   manually, extract the tag name (`build-N`) from the Location header.
2. **Compare** — parse N, compare with installed `longVersionCode`.
3. **Confirm** — if newer, ask user via `AlertDialog`.
4. **Download** — fetch APK directly via `HttpURLConnection` into
   `context.cacheDir`. Manual redirect handling (up to 5 hops). Progress
   shown as percent in the UI.
5. **Install** — `PackageInstaller` session, write APK bytes from cacheDir
   to the session, commit. The system install dialog appears.

## What was fixed and why

The earlier version used `DownloadManager` with
`setDestinationInExternalPublicDir(DIRECTORY_DOWNLOADS, ...)`. That landed
the APK in `/storage/emulated/0/Download/birdscope_update.apk`. On Android
11+ (Scoped Storage), the app cannot read its own APK back from the public
Download folder for the install step — fails with `EACCES (Permission denied)`.

Fix: bypass `DownloadManager` entirely. Download via `HttpURLConnection`
into `context.cacheDir` — app-private storage that the app always owns,
no extra permissions needed, no Scoped Storage friction. Side benefits:
manual progress reporting straight into `onStatus`, no system download
notification (which we did not need anyway).

## Trade-offs

- **No system notification** while downloading. UI status text is the
  only progress indicator. Acceptable: small APKs, foreground use.
- **APK in cacheDir** can be cleared by the system or by the user.
  This is fine: after install the file is no longer needed, and on next
  update the user re-downloads.
- **Manual redirect handling** because GitHub redirects from `releases/latest`
  to `releases/tag/build-N`, then again to the actual asset URL. Up to 5 hops
  is enough; never seen more than 2 in practice.

## Required permissions

- `INTERNET` — for fetching releases and APK.
- `REQUEST_INSTALL_PACKAGES` — for `PackageInstaller`. The user must
  also grant "Install unknown apps" via system settings; the updater opens
  that screen if `canRequestPackageInstalls()` returns false.

## Implementation

`Updater.kt`. Coroutine-based. Uses `MainActivity` callbacks `onStatus`
(text updates) and `onUpdateAvailable` (triggers the confirm dialog).
No external libraries.

## Status

Built and verified on device. First successful self-update through the
fixed code path: build with this fix installed cleanly via the previous
(broken) updater because that one only failed on read-back from Download,
and the fix was already in the new APK so the next cycle works clean.
