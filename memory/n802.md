# Purpose discipline

The `purpose` field on each link is the laziness mechanism of the memory system.
An LLM reads it and decides whether to descend. If `purpose` is unclear,
the LLM either skips a node it should read, or opens nodes it does not need.
Both are failures of navigation.

This leaf accumulates rules and observations specific to this project.
General principles belong in the spec (`protocols/project-memory.md`).

## Initial rules

### R1. Tell when to read, not what is inside

Bad: `purpose: "FFT analyzer"`
Good: `purpose: "details of FFT pipeline; read when changing window size, hop, or output format"`

The reader needs a trigger condition, not a label.

### R2. One sentence, ideally under 20 words

Longer purposes drift into content. If you need more, the content belongs
inside the leaf, not in the link description.

### R3. Do not duplicate the `name` field

Bad: `name: "UI", purpose: "the UI of the app"`
Good: `name: "UI", purpose: "windows, controls, menus; read when changing user-facing surface"`

The reader already saw `name`. Repeating it wastes the slot.

### R4. Mention the kind of work, not just the topic

Bad: `purpose: "decisions about audio"`
Good: `purpose: "open decisions about audio pipeline that need a choice before implementation"`

The reader filters by what they are doing, not only by domain.

## Open questions

- Should `purpose` ever reference other nodes by ID? (e.g. "see also n200")
  Probably no — that's what cross-branch links will be for once added.
- How often to revisit purposes for staleness? Currently: by hand, when noticed.

## Updates

New rules and observations are appended here as the project teaches them.
When a rule is mature enough to be universal, consider promoting it to the spec.
