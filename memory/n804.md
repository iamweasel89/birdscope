# Memory readability tests

Records attempts to read the projects memory from a fresh LLM session
with no prior context. The point is to measure whether the design
(`memory/` + README pointers + protocols) actually works in the wild.

These are not proofs — they are probes. Each probe surfaces specific
gaps or confirms specific guesses.

## 2026-05-02: Claude, single attempt

**Distinctive behaviour:**

1. Tried `github.com/blob/...` URL — failed.
2. Tried `raw.githubusercontent.com` — also struggled briefly.
3. Did not give up — used `git clone` via shell to obtain the full repo locally.
4. From the clone, read README, root.json, n002.json, n203.md, and
   PhasePortraitView.kt directly.
5. Documented its reasoning out loud in the response itself.

**Answer quality:**

- Every parameter accurate.
- Full v1-v4 tuning history reproduced.
- Correct cross-reference to source code.
- *Beyond the leaf:* offered two concrete improvement ideas
  (adaptive peakFloor by background noise, fading trail like an
  oscilloscope afterglow). These came from understanding, not from
  reading — Claude reasoned from the design.

**What this confirms:**

- Claude is the reader profile this design was built for.
- A reader with multiple fetch strategies (HTTP fetch, git clone via
  shell) handles edge cases that single-tool readers fail on.
- A reader that cross-references memory and code produces stronger
  answers than one that relies on either alone.
- A reader that thinks beyond the leaf, after reading it, is a different
  category of value entirely.

## Conclusion of the series

Eight probes (or six, if counting only fetch-capable readers). Findings:

**The design works** when:
- The reader has fetch (and ideally also a shell to clone).
- `ref` values include extensions.
- The README points to `memory/root.json` and discourages shortcuts.
- Auxiliary files (overview, indexes) clearly identify their audience.

**The design does not work** when:
- The reader has no fetch tool (cannot be fixed from our side).
- The reader fabricates failure (cannot be fixed from our side).

**The design degrades gracefully** when:
- The reader is non-deterministic (DeepSeek attempt 4 vs 3 / 5 / 7).
- Mitigations help but cannot fully prevent shortcuts.

**The design rewards careful readers**:
- Claude pulled both memory and code, and produced a substantive answer
  plus design suggestions. The structure invites this — purposes guide,
  leaves contain real content, code is accessible alongside.

**Lessons that crystallized into the spec / project rules:**

1. README defers state to `memory/`, never duplicates it.
2. `ref` always includes extension.
3. Auxiliary files declare audience explicitly.
4. Critical parameters live in the relevant leaf in stable places
   (frontmatter and first paragraph), not only in commits.

## What remains open

Future probes (no urgency):

- Long-form probe: walk the entire tree, summarize everything.
- Re-run the same prompt periodically to track stability over time.

Series 2026-05-02 closed with a clear positive result for the design
when read by a careful, fetch-capable LLM.

## 2026-05-03: Claude on claude.ai, meta-work probe

Different shape from the 2026-05-02 series: operator asked to read two
URLs (`birdscope/memory/root.json`, `protocols/launch-block.md`) and
moved straight into discussion of the memory concept itself, not into
product features.

**Stumbles surfaced:**

1. Fastly negative-cache hit on
   `raw.githubusercontent.com/.../protocols/main/launch-block.md`. CDN
   returned 404 for the full session despite the file existing.
   `github.com/.../blob/...` returned 200; rawLines extracted from the
   blob HTML via shell.

2. `web_fetch` rejects constructed URLs. Walking from `root.json` to
   `n00x.json` children by composing the URL from the `ref` field
   failed: the tool only follows URLs already present in the
   conversation. Worked around via shell + blob-HTML.

3. Cross-repo discoverability gap: `birdscope/memory/root.json` does
   not point to `protocols/`. An LLM descending memory-only never
   sees the protocols repo. Currently mitigated by
   `birdscope/README.md`, but only if the LLM enters via README.

4. `protocols/README.md` was too thin to serve as a meta-entry-point
   ("work on the memory concept itself" sessions). 7 lines, 2 of 6
   protocols listed, no LLM-targeted section, no seed URLs. Fixed
   in-session.

**Things that worked:**

- `bash` to `github.com` (allowlisted) for blob-HTML extraction when
  raw fails.
- `birdscope/README.md`'s "Direct entry URLs" — correctly anticipated
  the constructed-URL failure mode for project files.

**Fixes applied in-session:**

- Rewrote `protocols/README.md` on the model of `birdscope/README.md`.
- Added Principle 9 (Seed URLs in README) to `project-memory.md`,
  promoting the practice from birdscope-local to spec.

**Notes:** self-probe by the same Claude that did the walking, written
on operator's request from inside the session. Not a third-party probe.

## 2026-05-03 (followup): three fresh-session probes after meta migration

After the day's commits (meta migration, README rewrite, fetching.md,
probe discipline added to spec), three fresh Claude sessions were run.

**Probe A — top-level URLs.** Operator asked a fresh session: "what do
you see in these two repos? Briefly." with `github.com/.../protocols`
and `github.com/.../birdscope`. Result: full success. Session entered
`protocols/README.md`, walked the index, identified birdscope as the
first active instance, descended into birdscope's memory correctly,
reported Build 28 from 2026-05-02, ветка `legacy`, etc. The new
meta-entry-point design works as intended.

**Probe B — design review.** Operator asked the same session for
thoughts on the memory architecture. Session described `purpose` as
a trigger condition correctly, identified fragility-of-purpose as the
load-bearing weak spot, named cross-cutting links as the first
expected extension. Also reported a *factual error*: claimed that
`birdscope/README.md` lacks a "Direct entry URLs" section. Verified
against ground truth — the section is on line 20 of 75. The session
confabulated absence with confidence. This is the case study for
the new "Probe discipline" rule in the spec: probes describe the
LLM's experience, not the file state, and require ground-truth
verification before commit.

**Probe C — original two raw URLs (the test that started everything).**
Operator pasted the original first-session URLs:
`raw.githubusercontent.com/.../birdscope/main/memory/root.json` and
`raw.githubusercontent.com/.../protocols/main/launch-block.md`.
Result: root.json fetched and summarized; launch-block.md returned
404 across multiple fresh sessions, including after a real
one-line content diff was committed and pushed to that file. This is
NOT a memory-design failure — it is external CDN flakiness on one
specific URL, and the broader research (see fetching.md) confirms
push-based cache invalidation on raw.githubusercontent.com is
unreliable as a class.

**Lesson.** The "two raw URLs" test is a different test than the
"top-level URL" test. The first depends on individual file accessibility
on a flaky CDN; the second depends on whether the README chain reads
cleanly. The second is what this design optimizes; the first is shot
noise. Future probes should distinguish them in the summary.

**Lessons crystallized:**

- `fetching.md` updated to correct the false claim that push-based
  bust works, and to add cache-busting query string + githack as
  alternative recoveries.
- Probe discipline rule (already in spec) demonstrated value within
  hours of being added: probe B would have been logged as fact
  without it.

## 2026-05-03 (probe 11): launch-block delivery, fresh-paste pitfall

Operator opened a fresh Claude session, gave it the protocols repo as
entry point, and a task to insert one paragraph into `project-memory.md`
and return a launch block.

**Worked:**

- Entered through `protocols/README.md`, descended to `project-memory.md`
  correctly. The single-URL entry-point design held.
- When `raw.githubusercontent.com/.../launch-block.md` returned 404
  (still stuck from earlier in the day), the fresh Claude switched to
  the github.com blob URL on its own and read the protocol there. The
  fetching.md techniques activated automatically — without operator
  hand-holding. First time this loop closed end-to-end.
- The launch block produced was syntactically correct, used
  `git rev-parse --show-toplevel`, idempotency guards, and BOM-less
  UTF-8 — all per `launch-block.md` discipline.

**Stumble:**

On paste from chat to PowerShell, the chat client autolinkified file
names and the `System.IO` namespace. `project-memory.md` became
`[project-memory.md](http://project-memory.md)`; `System.IO` became
`[System.IO](http://System.IO)`. PowerShell took both literally,
`ReadAllText` failed with FileNotFoundException, the rest of the block
aborted cleanly via the existing guard.

This is a new class of pitfall: the failure is in the *delivery channel*,
not in the LLM's output or in the launch-block protocol. The operator
noticed it in the trace and surfaced it before any wrong file was
written. No data lost.

**Fix applied:** `launch-block.md` Trap 3 added documenting the
autolinkifier failure mode, with mitigations on both LLM and operator
sides. Recommendation to wrap filenames in backticks survives most
autolinkers.

**Observation for the journal:** the design held — README chain worked,
fetching.md fallback fired automatically, idempotency guard saved data.
The only gap was outside the design's scope (the chat client's
rendering), and it now has a documented entry.
