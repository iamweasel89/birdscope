# Memory readability tests

Records attempts to read the projects memory from a fresh LLM session
with no prior context. The point is to measure whether the design
(`memory/` + README pointers + protocols) actually works in the wild.

These are not proofs — they are probes. Each probe surfaces specific
gaps or confirms specific guesses.

## 2026-05-02: ChatGPT, attempt 1 (no fetch)

Did not read the repo. Generic essay on classical phase portraits.
When asked, admitted it has no automatic browsing.

## 2026-05-02: ChatGPT, attempt 2 (with web search)

Claimed the repo "returns an empty/incomplete page without files." False.
Useful reminder that LLMs fabricate process to avoid admitting failure.

## 2026-05-02: DeepSeek, attempt 1

Read the repo. Found `window 4096` and `peakFloor 500` from a commit
message. Mostly correct theory, but:

- Said phase portrait is "not implemented" (mislead by stale README line).
- Read `tau 3` instead of `tau 32` (hallucination).
- Wrong package name (`com.weasel.birdscope`).

**Fix applied:** README updated to defer feature state to `memory/`.

## 2026-05-02: DeepSeek, attempt 2 (after README fix)

Walked the tree, but tried `memory/n203.json` (file is `.md`). Recovered by
searching commits, found the first phase-portrait commit (`tau=8`), missed
the up-to-date n203.md leaf entirely.

**Fix applied:** explicit extensions on all `ref` values.

## 2026-05-02: DeepSeek, attempt 3 (after extension fix)

Full success. Walked: README -> root.json -> n002.json -> n203.md.
Answered correctly with all current parameters and the v1-v4 history,
pulled directly from the leaf body.

## 2026-05-02: DeepSeek, attempt 4

*Worse* than attempt 3 despite no design regression. Failed at root.json,
fell back to `overview.md` and used the bare `purpose` text plus invented
theory (`tau=1` was hallucinated). No further attempts to reach n203.md.

**What this revealed:** DeepSeek reading is non-deterministic. When it
failed at the proper entry point, it used overview.md as a shortcut.

**Fix applied:** README explicitly tells LLMs not to use overview.md as
the entry. overview.md itself opens with a banner pointing LLMs back to
root.json.

## 2026-05-02: DeepSeek, attempt 5 (after overview banner)

Full success again. Walked: README -> `memory/root.json` -> `n002.json` ->
`n203.md`. Read the leaf in full. Bonus: pulled `PhasePortraitView.kt`
and `MainActivity.kt` directly for cross-reference.

**Answer quality:** every parameter correct, history v1-v4 reproduced
faithfully, code references match. No hallucinations.

## Conclusion of the series

After five probes (one stalled tool, three successes, one stumble that
we fixed):

**The design works** when:
- The reader has a working fetch tool.
- `ref` values include extensions.
- The README points to `memory/root.json` and discourages shortcuts.
- Auxiliary files (overview, indexes) clearly identify their audience.

**The design does not work** when:
- The reader has no fetch tool. Cannot be fixed from our side; this is
  a property of the LLM tool, not the memory.
- The reader fabricates failure to avoid the work. Same.

**The design is robust but not bulletproof:**
- Even in good runs, readers vary in patience and method (some over-fetch,
  some under-read). Acceptable variance.
- A flaky reader can still cut corners — we mitigated by labelling
  shortcuts (overview banner), but cannot prevent every shortcut.

**Lessons that crystallized into the spec / project rules:**

1. README defers state to `memory/`, never duplicates it.
2. `ref` always includes extension.
3. Auxiliary files declare audience explicitly.
4. Critical parameters live in the relevant leaf in stable places
   (frontmatter and first paragraph), not only in commits.
