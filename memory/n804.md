# Memory readability tests

Records attempts to read the projects memory from a fresh LLM session
with no prior context. The point is to measure whether the design
(`memory/` + README pointers + protocols) actually works in the wild.

These are not proofs — they are probes. Each probe surfaces specific
gaps or confirms specific guesses.

## 2026-05-02: ChatGPT, attempt 1 (no fetch)

**Prompt:** "My project: https://github.com/iamweasel89/birdscope.
Help me understand how the phase portrait is structured — what parameters,
why these?"

**Outcome:** ChatGPT did not read the repo at all. Generic essay on
classical phase portraits in dynamical systems. When asked why it did not
follow the link, admitted it has no automatic browsing.

## 2026-05-02: ChatGPT, attempt 2 (with web search enabled)

**Same prompt, web search turned on.**

**Outcome:** Claimed the repo "returns an empty/incomplete page without
files" and refused to proceed. False claim — the repo is open. Useful
reminder that LLMs fabricate process to avoid admitting failure.

## 2026-05-02: DeepSeek, attempt 1

**Same prompt.**

**Outcome:** Read the repo. Found `window 4096` and `peakFloor 500` from
a commit message and produced a reasonable explanation of Takens-style
delay embedding.

**Errors:**

1. Said "phase portrait is not implemented" — false. Caused by a stale
   "Current build: minimal app — start screen only" line in the root README.
2. Read `tau 3` instead of `tau 32`. Hallucinated detail.
3. Used package `com.weasel.birdscope` instead of correct
   `com.iamweasel89.birdscope`.

**Fixes applied:** README updated to defer feature state to `memory/`.

## 2026-05-02: DeepSeek, attempt 2 (after README fix)

**Same prompt.**

**Outcome:** Walked the memory tree the way it was designed:

1. README → `memory/`.
2. `memory/root.json` → three branches.
3. `memory/n002.json` → three audio leaves.
4. Tried `memory/n203.json` — failed (file is `.md`, not `.json`).
5. Recovered by searching commits, found `PhasePortraitView.kt` directly.
6. Answered with code-level details from the **first** phase-portrait
   commit (`tau=8`), missing the up-to-date n203.md leaf entirely.

**Fix applied:** added explicit extensions (`.json`/`.md`) to all `ref`
values in branch JSONs. Spec updated.

## 2026-05-02: DeepSeek, attempt 3 (after extension fix)

**Same prompt.**

**Outcome — full success:**

1. README → `memory/`.
2. `memory/root.json` → branches.
3. `memory/n002.json` → audio leaves.
4. `memory/n203.md` → reached on first try (extension was explicit).
5. Answered correctly: window 4096, tau 32, decimation step 4, EMA 0.85,
   peak floor 500, full v1-v4 tuning history pulled straight from the leaf.
6. After answering, kept trying to fetch source via GitHub API for extra
   verification, hit a step limit. Slightly wasteful but did not affect
   the answer.

**What this confirms:**

- Memory **is readable** by an LLM with fetch tools and `purpose`-driven
  navigation works exactly as designed.
- Explicit extensions on `ref` values are essential — without them the
  reader either gives up or guesses wrong.
- Leaf body content (parameters, history, design notes) reaches the reader
  faithfully when reachable.

**Behavioural note (about DeepSeek, not the design):**

DeepSeek is greedy with sources — it will keep fetching even after it has
a complete answer in hand. Burns steps unnecessarily but does not corrupt
the answer. Cannot be fixed from our side.

## What to do with these results

The pattern after four probes:

1. README must not duplicate state that lives in `memory/`. (Fixed.)
2. `ref` values must include the extension. (Fixed.)
3. Critical parameters belong in the relevant leaf, not only in commit
   messages — confirmed working when leaf is reachable.
4. The design works for LLMs with fetch tools. ChatGPT-class tools without
   reliable fetch cannot use this memory regardless of how well it is built.
