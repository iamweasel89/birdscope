# Memory readability tests

Records attempts to read the projects memory from a fresh LLM session
with no prior context. The point is to measure whether the design
(`memory/` + README pointers + protocols) actually works in the wild.

These are not proofs — they are probes. Each probe surfaces specific
gaps or confirms specific guesses.

## 2026-05-02: ChatGPT (free tier, no fetch)

**Prompt:** "My project: https://github.com/iamweasel89/birdscope.
Help me understand how the phase portrait is structured — what parameters,
why these?"

**Outcome:** ChatGPT did not read the repo at all. It responded with
a generic essay on classical phase portraits in dynamical systems
(dx/dt, dy/dt, attractors, saddle points). When asked why it did not
follow the link, it admitted it has no automatic browsing.

**What this means:** not a defect in our memory. A limitation of the
LLM tool: ChatGPT in default mode cannot fetch URLs.

**Note:** ChatGPT also hallucinated process — it claimed to have
"checked and the repo does not exist publicly", which is false.
Useful reminder that LLMs invent process when they have no real access.

## 2026-05-02: DeepSeek (had fetch tool)

**Same prompt.**

**Outcome:** DeepSeek did read the repo. Found correct parameters
(window 4096, peakFloor 500) and explained them with mostly correct
theory (Takens embedding, time-delay reconstruction).

**Errors observed:**

1. Said "phase portrait is not implemented" — false. It is implemented
   and described in n203 as `status: done`. Likely cause: DeepSeek
   read the old "Current build: minimal app — start screen + Check
   for update button" section in the root README, took it as current,
   and did not check `memory/`.

2. Read `tau 3` instead of `tau 32`. Either hallucination or OCR-style
   misread of commit message text. Our n203 leaf clearly says `tau = 32`.

3. Used package name `com.weasel.birdscope` instead of the correct
   `com.iamweasel89.birdscope`. Minor but symptomatic of careless reading.

4. Framed `tau` as Takens reconstruction of an unknown attractor. Our
   case is simpler: known signal (audio), `tau` is just a delay. The
   theory is correct but slightly miscontextualized.

**What this means:**

- Memory **is readable** by an LLM with fetch tools. Confirmed.
- The design works: structure, descent by purpose, leaf details.
- **But:** the README had stale status text that misled the reader.
  Fixed in the same commit as creating this leaf — README now explicitly
  says it does not describe current features and points to `memory/`.
- LLMs can still hallucinate details even when reading correctly.
  Critical parameters should be repeated in multiple places for redundancy.

## What to do with these results

Each test surfaces something. The pattern so far:

1. README must not duplicate state that lives in `memory/`. The single
   source of truth principle applies even to the entry README.
2. Critical parameters in code (tau, window, etc.) should appear in
   the relevant leaf in a stable place (frontmatter or first paragraph),
   not only in commit messages.
