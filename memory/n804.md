# Memory readability tests

Records attempts to read the projects memory from a fresh LLM session
with no prior context. The point is to measure whether the design
(`memory/` + README pointers + protocols) actually works in the wild.

These are not proofs — they are probes. Each probe surfaces specific
gaps or confirms specific guesses.

## 2026-05-02: ChatGPT, attempt 1 (no fetch)

**Prompt:** "My project: https://github.com/iamweasel89/birdscope.
Help me understand how the phase portrait is structured — what parameters,
why these?"

**Outcome:** ChatGPT did not read the repo at all. Generic essay on
classical phase portraits in dynamical systems (dx/dt, dy/dt, attractors).
When asked why it did not follow the link, admitted it has no automatic browsing.

## 2026-05-02: ChatGPT, attempt 2 (with web search enabled)

**Same prompt, web search turned on.**

**Outcome:** ChatGPT claimed the repo "returns an empty/incomplete page
without files" and refused to proceed. False claim — the repo is open
and contains a full README, `memory/`, and source. Useful reminder that
LLMs will fabricate process to avoid admitting failure.

## 2026-05-02: DeepSeek, attempt 1

**Same prompt.**

**Outcome:** Read the repo successfully. Found `window 4096` and
`peakFloor 500` from a commit message and produced a reasonable explanation
of Takens-style delay embedding.

**Errors:**

1. Said "phase portrait is not implemented" — false. Likely cause: read
   stale "Current build: minimal app — start screen only" line in the root
   README and took it as current state.
2. Read `tau 3` instead of the correct `tau 32`. Hallucination of detail.
3. Used package `com.weasel.birdscope` instead of `com.iamweasel89.birdscope`.

**Fix applied:** README updated to explicitly say it does not describe
current features and to defer to `memory/`.

## 2026-05-02: DeepSeek, attempt 2 (after README fix)

**Same prompt.**

**Outcome:** Walked the memory tree the way it was designed:

1. Read root README.
2. Read `memory/root.json` — saw the three branches.
3. Read `memory/n002.json` — saw the three audio leaves.
4. Tried `memory/n203.json` — failed (file is `.md`, not `.json`).
5. Recovered by searching commits, found `PhasePortraitView.kt`
   in commit `a20e70c` and read it directly.
6. Answered with code-level details: PHASE_SIZE = 1024, tau = 8, scaling.

**What worked:**

- Memory navigation by `purpose`: confirmed.
- LLM with fetch tool descends correctly through the tree.
- Self-recovery when the navigation broke (searched commits, found the code).

**What broke:**

- `ref: "n203"` without extension forced the LLM to guess `.json`. It failed,
  recovered, but the leaf body (which contains the up-to-date parameters and
  tuning history) was never reached.
- Result: the answer cited stale parameters from the first commit on phase
  portrait (`tau=8`) rather than the current values from n203.md (`tau=32`,
  decimation step 4, peak floor 500).

**Fix applied:** added explicit extensions to all `ref` values in branch
JSONs (`n201.md`, `n800.json`, etc.). Spec updated in protocols repo to
require this.

## What to do with these results

Each test surfaces something. The pattern so far:

1. README must not duplicate state that lives in `memory/`. Single source
   of truth applies even to entry README. (Fixed.)
2. `ref` values must include the extension; otherwise readers guess wrong
   and miss leaves. (Fixed in this commit.)
3. Critical parameters in code (tau, window, etc.) should appear in a
   stable place inside the relevant leaf, not only in commit messages.
   Already true for n203 — the issue was that the leaf was unreachable,
   not that the leaf was wrong.
