# Memory readability tests

Records attempts to read the projects memory from a fresh LLM session
with no prior context. The point is to measure whether the design
(`memory/` + README pointers + protocols) actually works in the wild.

These are not proofs — they are probes. Each probe surfaces specific
gaps or confirms specific guesses.

## 2026-05-02: ChatGPT, attempt 1 (no fetch)

Did not read the repo. Generic essay on classical phase portraits.
When asked, admitted it has no automatic browsing.

## 2026-05-02: ChatGPT, attempt 2 (with web search)

Claimed the repo "returns an empty/incomplete page without files." False.
Useful reminder that LLMs fabricate process to avoid admitting failure.

## 2026-05-02: DeepSeek, attempt 1

Read the repo. Found `window 4096` and `peakFloor 500` from a commit
message. Mostly correct theory, but:

- Said phase portrait is "not implemented" (mislead by stale README line).
- Read `tau 3` instead of `tau 32` (hallucination).
- Wrong package name (`com.weasel.birdscope`).

**Fix applied:** README updated to defer feature state to `memory/`.

## 2026-05-02: DeepSeek, attempt 2 (after README fix)

Walked the tree, but tried `memory/n203.json` (file is `.md`). Recovered by
searching commits, found the first phase-portrait commit (`tau=8`), missed
the up-to-date n203.md leaf entirely.

**Fix applied:** explicit extensions on all `ref` values.

## 2026-05-02: DeepSeek, attempt 3 (after extension fix)

Full success. Walked: README -> root.json -> n002.json -> n203.md.
Answered correctly with all current parameters and the v1-v4 history,
pulled directly from the leaf body. Slight time waste at the end (kept
fetching code via GitHub API for extra verification), but did not affect
the answer.

## 2026-05-02: DeepSeek, attempt 4

Same prompt, fresh session.

**Outcome:** *worse* than attempt 3 despite no design regression.

1. README -> tried `memory/root.json` -> Read failed.
2. Did not retry as raw.githubusercontent.com.
3. Searched the web for "phase portrait birdscope". Got noise.
4. Eventually tried `memory/overview.md` -> succeeded.
5. Used **overview.md as the only source**.
6. Answered with the bare `purpose` text from overview ("delay-embedded
   2D scatter visualization") plus invented theory: claimed `tau=1`
   (false — actual is 32) and "2D embedding from Takens".

No further attempts to reach n203.md were made.

**What this confirms (and what it does not):**

- The design did not regress. It worked in attempt 3 with the same files.
- DeepSeek reading is **non-deterministic**. Two runs produced different
  paths through the tree. We cannot rely on every run reaching the leaf.
- When DeepSeek failed at root.json, it fell back to overview.md and
  treated it as a sufficient summary. This is **wrong but predictable**:
  overview.md looks like an index, so the LLM uses it that way.

**Fix applied (in this commit):**

- README explicitly tells LLMs not to use overview.md as the entry.
- overview.md itself opens with a banner: "audience: human operators",
  pointing LLMs back to root.json.

These are mitigations, not solutions. A persistent flaky reader will still
cut corners. The honest takeaway: even a well-built memory needs a reader
with patience, and reader patience varies.

## What to do with these results

The pattern after five probes:

1. README must not duplicate state that lives in `memory/`. (Fixed.)
2. `ref` values must include the extension. (Fixed.)
3. Critical parameters belong in the relevant leaf, not only in commits.
   (Already true; works when leaf is reached.)
4. Auxiliary files like `overview.md` must signal that they are not the
   entry point, otherwise an LLM may use them as a shortcut. (Fixed.)
5. Some LLM behaviour is non-deterministic. The same design produces
   different reads on different runs. Acceptable failure mode: the
   design works for a careful reader, not for every reader.
